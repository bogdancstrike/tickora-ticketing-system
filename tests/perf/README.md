# Tickora performance suite (k6)

Scripts that exercise the hot read/write endpoints against a seeded
backend. The 30M-ticket seed is generated by `scripts/seed_dev_30M.py` —
run that first to get a representative dataset.

## Running

```bash
# Auth: scripts assume the seeded `bogdan` (full platform access) user.
# Pull a token once and export it; the scripts read it from $TICKORA_TOKEN.
export TICKORA_TOKEN=$(scripts/k6_token.sh)

k6 run tests/perf/tickets_list.js
k6 run tests/perf/monitor_overview.js
k6 run tests/perf/admin_overview.js
k6 run tests/perf/ticket_create.js          # write path; rate-limited
```

Tune the `BASE_URL` env var if running against staging:

```bash
BASE_URL=https://tickora.staging.example k6 run tests/perf/tickets_list.js
```

## What each script measures

| Script | Endpoint | Goal |
|---|---|---|
| `tickets_list.js` | `GET /api/tickets` | p95 < 700 ms with the partial indexes from `9a1f3e0c2d10`. |
| `monitor_overview.js` | `GET /api/monitor/overview?days=30` | p95 < 1.5 s. The 60-s Redis cache should bring the second hit under 100 ms. |
| `admin_overview.js` | `GET /api/admin/overview` | p95 < 1.5 s. Active-session KPI involves a Redis SCAN — verify under load. |
| `ticket_create.js` | `POST /api/tickets` | Rate limiter (`RATE_LIMIT_TICKET_CREATE_PER_MIN`) should kick in around 20/min/user. |

Thresholds and stage shapes are tuned for a 200–300 concurrent user target
per the BRD.
